## 1. サービス概要

**weight_dialy は「ずぼらな人」が、入力ゼロで「自分も動ける側だ」と気付けるようになる運動アプリです。**

「健康のためにスマートウォッチでも買うか…と思ったら、**そんな金はねぇ！！** となった」 — そんな開発者本人が、自分のスマホがすでに歩数や階段昇降を勝手に計測していることに気付き、それを拾って褒めてくれるアプリとして作り始めました。

スマホが既に計測している歩数・距離・階段を Apple Shortcuts 経由で自動取り込み、AI が「今日◯◯食べてもプラマイゼロ」と日々提案します。「ジムに行かないけど何かしたい」人の最初の一歩を、入力作業ゼロで支えます。

**誰の、どんな課題を、どう解決するか**

- **誰**: ジムにも筋トレにも行かない通勤通学カジュアル層 (25〜40 歳、ハイブリッド勤務、都市部・公共交通通勤)
- **どんな課題**: 「自分は運動していない」というセルフイメージで罪悪感ループに陥り、実際には毎日歩いたり階段を上っているのに、自己肯定の機会を失っている
- **どう解決**: スマホが既に計測している運動データを Apple Shortcuts 経由で自動取り込み → 入力ゼロで日常活動を可視化。AI 提案・カロリー貯金システム (将来) で「動ける側の人間」のセルフイメージを獲得させ、本格運動への自然な踏み出しを促す

## 2. このアイデアはどこから生まれたか

### 2-1. きっかけとなった体験・感情

私はジムに行ったりランニングしたり、そういう運動を一切してきませんでした。日常的に暴飲暴食を繰り返し、こんなに大きくなりました。

最近は罪悪感が強く、好きなはずのラーメンもなかなか食べられません。

ただ、よくよく考えてほしいんです。寝たきりでない限り、我々は何らかの形で運動をしています。たまの出社の帰りに 3 駅余分に歩いたり、地下鉄ホームまでの長いエスカレーターを使わず階段で上がったり。

そういう「日常のプラス α」のたびに、私は心の中で「ちょっとはカロリー消費できたかな?」と、自分を少しだけ褒めたくなる気持ちが湧いてきます。

ある時 「これを記録できたら面白いな、スマートウォッチでも買うか」 と考えたのですが、よく値段を見て **「そんな金はねぇ！！」** となりました。仕方なくスマホで何ができるか調べてみたら、**iPhone も Android もすでに歩数や階段昇降を勝手に計測している** ことを知りました。なんだ、わざわざ高い時計を買わなくても、自分のデータはもう手元にあるじゃないかと。

そこから「**このデータを拾って褒めてくれるアプリを作れば、自分のような人が救われるんじゃないか**」と思い至ったのが、weight_dialy の出発点です。

### 2-2. なぜそれが「気になった」のか

同じ体験をしても気にしない人はいると思います。なぜ私が気になったのかを振り返ると、**2 つの自分の性質** に行き着きます。

ひとつは、**罪悪感ベースで自分を律することの限界**を知っているからです。「ラーメン食べちゃダメ」と禁止のかたちで自分を制御しても続きません。続くのはむしろポジティブな自己肯定の方で、「今日も小さな勝ちがあった」と褒められる回路がほしかった。

もうひとつは、**「自分以外にも同じ気持ちの人がいるはず」という強い共感**です。フィットネス勢向けのアプリは世の中に山ほどあるけれど、ジムに行かない大多数の人 — 通勤通学のついで歩きや階段選択を **「これも運動だ」と認めてくれる仕組み** がない。それが気になって仕方ありませんでした。

## 3. 課題の整理（表面的になっていないか）

### 3-1. 表に見えている困りごと

自分は運動なんてしていないと思っていて、暴飲暴食の罪悪感で食事も楽しめない。「ジムに行く / ランニングする」みたいな大きな運動をしないと、自分の体への投資はゼロだと思い込んでいる。

### 3-2. 本当に解決したい課題は何か (5 whys)

**Q1**: なぜ「自分は運動していない」と思い込むと困る?
**A1**: 罪悪感が積もって食事も楽しめず、生活の質が下がる。

**Q2**: なぜ罪悪感ベースの自己制御だと QOL が下がる?
**A2**: ネガティブモチベは続かない。「我慢」では反動が来る。

**Q3**: なぜ続くモチベがないと困る?
**A3**: 「動いていない自分」というセルフイメージが固定化される。

**Q4**: なぜセルフイメージ固定化が問題?
**A4**: 実際は日常で歩いたり階段を使ったりしているのに、それが「運動」として認識されないため、 **「自分は動ける側の人間だ」という小さな自信の芽が育たない**。

**Q5**: なぜその「自信の芽」が必要?
**A5**: 自信の芽があれば、いつか **「ちょっと走ってみるか」「ジムも見学だけしてみるか」** という、本格運動への自然な踏み出しが起こる。**逆にそれがなければ、永遠に「自分には無理」のままでループする**。

→ **本当に解決したい課題**:

> **「ジムに行かない人」が、自分の日常の動き (歩く / 階段) をデータで認識して、「動ける側の人間だ」というセルフイメージを少しずつ獲得していくこと。その積み重ねを、いつか「ちゃんとした運動」への自然な第一歩につなげること。**
>
> **weight_dialy はその「入り口」「最初の一歩」のための装置である。**

## 4. 想定ユーザーについて

### 4-1. 想定しているユーザー

**年齢設定**
- 年齢: **25 歳〜40 歳** (働き盛りで運動習慣が抜け落ちた層)
- 理由: 学生時代は部活等で動いていたが、就職後 5〜15 年で運動から遠ざかり、暴飲暴食が始まった頃。「健康診断で引っかかる前にどうにかしたい」意識は芽生えるが、ジムに行く時間/お金/モチベがまだ揃わない世代。10 代は親の管理下、50 代以降は健康問題が深刻化してから本格対策を始めるため、**「入り口装置」としての価値が最も効くのがこの年代**。

**生活・状況設定**
- 生活: **平日は在宅勤務 + たまの出社 (週 1〜2 回) のハイブリッド**。デスクワーク中心。
- 理由: 完全リモートだと通勤の "ついで歩き" 自体が発生しないので、本アプリの主軸データ (歩数 / 階段) が拾えない。完全出社だと既に運動量が一定確保されている可能性がある。**ハイブリッド層こそ「出社日のついで歩き」が運動量の最大変動要因** で、ここを可視化する価値が高い。

**環境設定**
- 環境: **都市部 (東京等)、公共交通機関ベースの通勤**
- 理由: 車通勤だと「歩く / 階段」の選択肢自体が日常に存在しない。**駅利用 (= エスカレータ vs 階段の選択、1 駅前下車などの裁量) があるユーザー** こそ「日常のプラス α」が積み上がる。Section 2 で挙がった「地下鉄ホームまでの長い階段」のような原体験はこの環境に固有。

### 4-2. 自分とユーザーの距離

→ **自分自身** (現在進行形でこの問題を抱えている)

理由: README Section 2 の通り、自分自身が「ジムに行かないけど暴飲暴食 → 罪悪感」ループに入っており、自分のためのアプリを作っている。**N=1 ユーザーリサーチが完璧に成立する** (=仕様判断に迷ったら「自分が使いたいか」で決められる) のが最大の強み。

加えて **「過去の自分」** の要素もある — 学生時代は運動していたので「動ける側の人間」だった感覚を覚えている。**その感覚を取り戻したい現在の自分** と **過去の自分** を繋ぐ装置としての位置付け。

### 4-3. 実際に使われる可能性について(需要・存在確認)

**Q1: まず最初に使ってくれそうな人は誰か?**

→ **スクールの同期 (プログラミング学習仲間)** が筆頭。発表会で触ってもらえる N=10〜50 の母集団がいる。20〜30 代のエンジニア志望、デスクワーク中心、ハイブリッド勤務 = 想定ユーザー像とほぼ重なる。次点で **自分自身** (継続ユーザー候補)、**SNS でコンセプトに共感した知人**。

**Q2: どんな場面でアプリを思い出すか?**

→ 主に 2 つの瞬間:
- **「ラーメン食べたいけど罪悪感」** と感じた時に開く (= AI の「プラマイ提案」を見る用途)
- **エスカレータと階段の分かれ道** で迷った時、「今日の Streak 切らしたくない」が後押しになる (= 行動の瞬間に効くゲーミフィケーション)

**Q3: 今の生活の中で代わりに使っているものは?**

| 既存ツール | 何ができてるか | 何が足りないか |
|---|---|---|
| iPhone ヘルスケア (純正) | 歩数 / 階段 自動計測 | **「今日のあなたを褒める」体験が皆無**、グラフだけで終わる |
| Apple Watch | 同上 + リング閉じる演出 | 専用デバイス必要 (¥50,000+)、フィットネス勢向け |
| Fitbit / Garmin | 同上 | 同様にデバイス必要、フィットネス勢向け |
| **何もしていない** (= 大多数) | — | 「動いていない自分」のセルフイメージが固定化 |

**Q4: それを置き換えてまで使う理由は?**

→ 既存ツールは「データを見せる」だけだが、weight_dialy は **「データを根拠に褒める / AI が今日の食事提案する」** ところまで踏み込む。**「測定 → 評価 → 行動提案」のループを最小手数で完結させる** のが他にない価値。Apple ヘルスケア + ChatGPT を毎日手動で繋ぐ作業を、1 アプリで自動化したものとも言える。

### 4-4. ユーザーがサービスを導入して利用するまでのイメージ

**(1) どんな場面でアプリを知る・思い出すか**
- スクールの発表会で実物を触る → 同期からの口コミ伝播
- SNS (X / Zenn) で開発者本人が「自分の Streak」を発信して認知拡大
- 「ラーメン食べていいかな?」と検索した時、AI 提案アプリの選択肢として浮上

**(2) 最初の一歩 (登録・導入)**

> **初回設定こそが唯一の山。逆に言えば、ここを越えれば一生ゼロ操作。** 「ずぼらな人向け」というポジションは継続フェーズで成立するもので、初回 5 分の山だけは越えてもらう必要がある (= 全自動アプリの宿命)。

1. Google でログイン (ワンタップ)
2. iPhone ユーザー: 設定画面で「Apple Shortcuts 連携」をタップ → token コピー → iCloud で配布されている事前作成済み Shortcut を追加 → 完了 (合計 5 分)
3. Android ユーザー: 自動的にデモモード。「Android 版は鋭意制作中」バナーで体験のみ

**(3) 利用を続ける理由 (習慣・必要性・価値)**
- **🔥 Streak (連続記録)**: 切らしたくないという心理が日々のアプリ起動を促進
- **AI のプラマイ提案**: 毎日違う食べ物提案が出るので「今日は何食べていいかな?」と確認しに来る
- **「動ける側の人間」セルフイメージの蓄積**: 30 日チャートの右肩上がりが見える瞬間にポジティブ強化
- **罪悪感ベースじゃない**: 我慢を強要しないので反動が起きにくい
- **🆕 (将来構想) カロリー貯金システム**: 日々の消費カロリーを「貯金」として累積、マイルストーン到達時に「**おめでとう！ラーメン 1 杯分のカロリーを消費できたよ！**」バナー通知。食べた分は貯金から差し引く財務感覚で管理 → 罪悪感を「赤字 / 黒字」に置き換える。詳細は Section 9 参照

## 5. 既存サービス・競合調査

### 5-1. 似たサービスの調査

- **サービス名**: あすけん
- **URL**: https://www.asken.jp/
- **何ができるか**:
  - 株式会社 asken 運営、iOS / Android アプリ + Web
  - **食事の写真撮影 → AI が栄養素自動解析** (写真認識を売りにしている)
  - 1 日の摂取カロリー / PFC バランスを記録
  - **AI 栄養士「未来」が日次アドバイス**
  - 運動量・体重も記録可能 (基本手入力)
  - 無料 + プレミアム月額課金 (¥480/月) でアドバイス精度や履歴閲覧期間が拡張
  - **国内ユーザー数 1,000 万超**、食事記録カテゴリの代表的存在

### 5-2. それでも自分が作りたい理由

私自身が **過去あすけんを試して 1 日で挫折した経験** があります。理由はシンプルで、**登録する手間がもうほんとに嫌いで嫌いで**、初日で続けられなくなりました。

食事の写真を撮る、食材を選ぶ、分量を入力する — この作業を 1 日 3 回やると考えた時点で「**こんなのダルすぎて絶対に続かない**」となり、たった 1 日でアプリを開かなくなりました。

振り返ってみると、私が本当に欲しかったのは **「毎食記録すること」ではなく、「自分の生活が改善されているかを楽に確認できる仕組み」** でした。

あすけんは「摂取側の正確な記録」を主軸に置く設計です。これは管理栄養士的なアプローチとして合理的なのですが、**「ジムに行かないし食事も気軽に楽しみたい一般人」にとっては入力の手間が大きすぎ** 、私のように初日で挫折する人が一定数いるはずです。

自分のような **「気軽に運動を意識し始めた段階」のユーザー** に必要だったのは、**「最初に 1 回設定すれば、あとは何もしなくても勝手にデータが溜まる」** 仕組みです。スマホは元から歩数や階段昇降を勝手に計測しているのだから、それを Web 側に流し込めば、記録のための手作業は限りなくゼロにできるはず。

**入力ゼロで継続できるからこそ、最初の一歩を踏み出した人が脱落せず、データが溜まり、自己肯定の回路が回り出す**。これが weight_dialy を作りたい根本的な理由です。

### 5-3. 差別化を一文で決める

> **「ずぼら」な人のための、入力ゼロで自己肯定が回り出す運動アプリ**

## 6. このサービスで提供したい価値

### 6-1. ユーザーの変化

**行動の変化**
- エスカレータと階段の分かれ道で、**「Streak 切らしたくない」を理由に階段を選ぶ**ようになる
- 出社日に「**今日は 1 駅前で降りるか**」が選択肢として浮かぶ
- 食事の前に **「今日のプラスα消費」をアプリで確認** してから注文する習慣が芽生える

**気持ちの変化**
- 「ラーメン食べたい → 罪悪感」 → 「ラーメン食べたい → アプリで確認 → **データに基づいて OK と判断 → 罪悪感なく食べる**」
- 「動いていない自分」への自己嫌悪 → 「**実は毎日動いてた、自分悪くないじゃん**」という発見

**考え方の変化**
- 「**運動 = ジム or ランニングでないと意味がない**」 → 「**日常の小さな選択も全部運動にカウントされる**」
- 「**記録は手間**」 → 「**記録は勝手に溜まる**」 (= ずぼらな自分でも継続できる、という自己効力感)
- 中長期的には: 「自分には無理」 → 「**ちょっと走ってみるか**」「ジムも見てみるか」 (= 本格運動への自然な踏み出し)

### 6-2. 価値を一文で表す

> このサービスは **「ずぼらな人」** が、入力なしで **「自分も動ける側だ」と気付ける** ようになるサービス。

## 7. このアプリで実現すること

### MVP で作る機能

- ✅ Google ログイン
- ✅ Apple Shortcuts → Webhook 受信 (歩数 / 距離 / 階段)
- ✅ Sketchy 風 UI テーマ
- ⏳ 30 日 消費カロリーチャート
- ⏳ 🔥 Streak (連続記録)
- ⏳ AI プラマイ提案 (今日◯◯食べてもプラマイゼロ)
- ⏳ 設定画面 (Apple Shortcuts 連携 token 表示 / コピー)
- ⏳ Android デモモード (制作中アピール)

### 本リリースで作る機能

#### カロリー貯金システム

- 日々の消費カロリーを「貯金」として累積
- マイルストーン到達 (ラーメン1杯分など) で祝福バナー
- 食べた分は貯金から差っ引く → **罪悪感を「赤字 / 黒字」の財務感覚に置き換える**
- アプリの哲学を機能として完成させる目玉 (詳細仕様は Section 9 のロードマップ参照)

#### ガチ運動モード — 本格運動に踏み出した人向けの解禁機能

- 自転車距離の取り込み (`cycling_distance_meters` 追加)
- ワークアウト記録 (ランニング / 水泳 / 筋トレ等の HKWorkout 取り込み)
- 心拍数記録 (Apple Watch 連携)
- 体重記録 + 消費との関係分析 (= **アプリ名 weight_dialy の本来の意味回収**)
- (将来) Strava 連携で能動記録勢の世界に橋渡し

#### プラットフォーム拡張

- **Capacitor** (= Web アプリをネイティブシェルで包んでスマホアプリ化する OSS フレームワーク) + **Health Connect** (= Android のヘルスデータ統合 OS 機能、HealthKit の Android 版) で **Android 完全対応**

> **コンセプト**: 「ずぼらモード」(MVP) で入り口に立った人が、自然と「ガチ運動モード」を解禁したくなる導線を作る。

## 8. このアプリの懸念点とその対策

### 懸念点①: iPhone 偏重ゆえの市場リーチ問題 (兼: 開発者自身が使えない問題)

**何が問題になりそうか**
- スマホシェアで Android は世界 70%・日本 30%+。MVP では Android ユーザーがデモモードしか使えない
- **そもそも私自身が Android ユーザー** であり、「ずぼらな自分のために作る!」と意気込んで開発を始めたのに、 **気づいたら自分が一番使えない人になっていた** という構造的なトホホを抱えている
- ドッグフードできない = 発表会後の継続改善で「自分が日々使ってるからこの機能足したい」というフィードバックループが回らない

**なぜそう思うか**
- 技術的制約の調査結果:
  - Health Connect は Web から直接アクセス不可 (Android SDK 専用)
  - Google Fit REST API は 2024-05 以降新規プロジェクト登録不可
  - Tasker / MacroDroid 等は有料 + 設定難易度高くカジュアル層に届かない
- 4 日の MVP 期間内で Android 自動連携を実装する公式手段がないと判明し、デモモードで割り切る判断に至った

**そのために考えている対策**
- **短期 (発表会 = MVP 期間)**: デモモード + 「制作中」明示。スクール内部発表という限定オーディエンスなので、誠実にコミュニケーションすれば十分機能する想定
- **中期 (発表会後 = 本リリース v1.0)**: **Capacitor + Health Connect で Android ハイブリッドアプリ化を最優先タスク** に据える。これを v1.0 リリースの定義とする (= 開発者本人がドッグフードできる状態)
- **長期 (v1.x 以降)**: 開発者本人が日次で使い続けることで、本物のフィードバックループを稼働させる

## 9. 今後の展開・発展の方向性

### リリースロードマップ (3 段階)

| バージョン | スコープ | 優先理由 |
|---|---|---|
| **v1.0** | Capacitor + Health Connect で Android 完全対応 | 開発者本人がドッグフードできる状態を確保、改善ループの起点 |
| **v1.1** | カロリー貯金システム (Section 7 で予告した目玉機能) | アプリの哲学 (罪悪感 → 財務感覚) を機能で完成させる |
| **v2.0** | AI 提案ロジックの個別最適化 | データ蓄積後の差別化の核 |

minor 機能群 (PWA / ダークモード / ウィジェット / SNS / 各種ガチ運動モード機能) は上記の合間に随時取り込み。

### カロリー貯金システム (v1.1 詳細)

Section 7 で予告した目玉機能。仕様の核:
- 累計消費 - 累計摂取 = 貯金残高
- マイルストーン到達 (ラーメン 1 杯 ≈ 1000kcal、大盛りカツ丼 ≈ 1500kcal、ジョッキビール ≈ 200kcal) で祝福バナー
- 摂取側の入力は AI 提案を「食べた!」ボタンで承認した時点で自動引き落とし (= 入力ゼロ理念を維持)
- 残高がマイナスになったら「赤字警告」(= 罪悪感じゃなく財務感覚で警鐘)

### データが溜まった先のアイデア帳

- **個人ベースライン学習**: 「今日は普段比 +20% 動いた」のような相対評価
- **季節 / 曜日 / 天気別パターン分析**: 「梅雨は平均 -30% になる傾向」
- **AI 推定枠**: 「重い荷物の買い物」「坂道散歩」などを歩数 + 心拍データから推定して加点

### 他者との繋がり

- スクール内 / 友人内のストリーク比較 (競争ではなく共感型)
- カロリー貯金額のシェア機能 (「今週 ラーメン 2 杯分 貯まりました」を SNS シェア)

### 運用・仕組みで価値を出す

- 開発者本人 (Android ユーザー) のドッグフードを v1.0 リリースの定義とし、改善ループを構造的に保証
- 「今日のずぼら開発者の動き」を SNS 発信、アプリ自体が PR 媒体になる構造
- スクール卒業後も継続開発を公言、教材性を維持

## 10. 技術スタック（手段としての技術）

### 10-1. 使用予定の技術

**フレームワーク・言語**
- Ruby 3.4.9 / Ruby on Rails 8.1.3

**DB**
- PostgreSQL 16 (Dev Container 内)

**フロントエンド**
- Hotwire (Turbo + Stimulus) — SPA 化せずサーバ駆動
- Tailwind CSS 4 + daisyUI 5 — 標準コンポーネント快速
- **Sketchy custom CSS テーマ** (Claude Design 由来の手描き風)
- **Importmap** (= ビルドステップなしで JS モジュール管理する Rails 8 標準、esbuild / webpack 不使用)

**インフラ・運用**
- **Solid Trifecta** (= キャッシュ・ジョブ・ケーブルをすべて DB で賄う Rails 8 標準セット。Solid Queue / Solid Cache / Solid Cable) — 外部 Redis 等不要
- Kamal / Render / Fly.io から Day 5 で確定予定
- Dev Container (`.devcontainer/`)

**外部連携**
- Apple Shortcuts → Webhook (Bearer token 認証)
- (将来) Strava OAuth、Capacitor + Health Connect、OpenAI / Anthropic API

**主要 gem**
- `omniauth-google-oauth2` (認証)
- `has_secure_token` (Webhook トークン)
- `propshaft` (asset pipeline)
- `dotenv-rails` (開発時環境変数)
- `rspec-rails` + `factory_bot_rails` (テスト基盤、`capybara` + `cuprite` は将来導入予定)
- `rubocop-rails-omakase` (lint)

### 10-2. 技術選定の理由

**なぜこの技術を使うのか**

- **Rails 8 + Solid Trifecta**: 外部依存 (Redis 等) ゼロでデプロイが圧倒的に簡単。発表会向け短期間プロジェクトと相性◎
- **Hotwire**: SPA 化のオーバーヘッド回避。Stimulus + Turbo で十分インタラクティブ、Rails 標準で**教材性が高い**
- **Tailwind + daisyUI + Sketchy 共存**: 標準コンポーネントは daisyUI、世界観は Sketchy テーマで分離。書き換えコスト最小
- **Apple Shortcuts → Webhook**: ターゲット (iOS カジュアル層) と相性◎、ネイティブアプリ開発不要、初期実装コスト最小
- **Importmap**: ビルドステップなし、Rails 8 デフォルト。**「JS は最小限」の方針が初学者にも明示的**

**今回チャレンジしたい点**

- **Apple Shortcuts → Rails Webhook という新しい連携パターン** の実装と教材化
- **Sketchy 風カスタム CSS テーマと daisyUI の共存** (= 個性とコンポーネント快速の両立)
- **Capacitor + Health Connect** によるハイブリッドアプリ化 (本リリース後、学習別枠)
- **AI API による個別最適化提案** (将来、OpenAI or Anthropic)

**不安な点**

- **Capacitor + Android Health Connect 連携の経験ゼロ** → 学習コスト見積もり困難。本リリース以降に別枠学習期間を確保
- **AI API 連携経験ゼロ** → 同上
- **公開デプロイ + 監視運用が初めて**: Kamal / Render / Fly.io 選定 + 本番障害対応の知見不足

### 10-3. キャッチアップ不足の懸念

**MVP 期間中 (発表会まで)**: 採用技術はすべて Rails Way + Hotwire + 標準 gem の範囲。**キャッチアップ不足のリスクは小さい**。Apple Shortcuts は専用調査メモ (`memory/project_apple_shortcuts_webhook_research.md`) に集約済みで、迷ったらすぐ参照可。

**本リリース以降の懸念**:
1. **Capacitor + Health Connect SDK**: TypeScript / Kotlin の知識が必要。本リリース版に着手する前に **2〜3 週間の学習期間を別枠で確保** する想定。
2. **AI API 連携**: プロンプトエンジニアリング + コスト管理 + レイテンシ対策の総合的な学習が必要。試作版を別ブランチで小さく作ってから本体に統合する段階的アプローチを取る想定。

これらは MVP のスコープ外なので、**発表会には影響しない**。本リリース版開発時に、機能リリース計画と並行して学習計画を立てる予定。
