# /plan-issue

「1 Issue / 1 機能 / 1 PR」運用を支援するコマンド。
ユーザーが次に着手したいタスクを **対話で詰めて** から、必要に応じて GitHub Issue に変換する。

## 入力

トピック: $ARGUMENTS

ユーザーが軽くトピックだけ渡している場合は上に文字列が入る。空ならまずトピック自体を聞く。

## フロー

このコマンドは「設計を詰める対話」が主目的、起票は副産物。深い対話を惜しまない。

**対話開始時の宣言**: Step 3 に入る前に「このあと最大 4 つ質問しますが、自明な項目は省略します。途中で『もういい』と言えばそこで止めます」とユーザーに伝える (コントロール感を渡すため)。

### Step 1: トピック確認

- $ARGUMENTS が空 → 「次に何をやりますか? 短い言葉で OK です」と聞く
- $ARGUMENTS にトピックがある → 「『$ARGUMENTS』を詰めますね」と確認してから次へ

### Step 2: 背景を聞く (Step 3 の最初の質問を先取りする)

**重複チェックを先に走らせない**。ユーザーの意図を引き出す前に既存 Issue を見せると、loss aversion で意欲が萎える。

- 「なぜ今これをやるのか? 何が困っているか or 次のステップとして必要か」を聞く
- 回答を簡潔にメモして、Step 3 で使う

### Step 3: 重複チェック (背景を踏まえてから)

ユーザーが背景を語った後にこのチェックを実行することで、「やはり自分のはこういう文脈で違う」と自信を持って判断できる状態になる。

```bash
gh issue list --search "<トピック関連キーワード>" --state all --limit 5
```

- 似た Issue が見つかったら「#XX が近そうですが、別 Issue にしますか? それとも統合しますか?」と判断を仰ぐ
- 何も見つからなければ次へ進む

### Step 4: 残りを詰める (1 問ずつ、自明なら省略可)

以下を **1 問ずつ** 聞く。Claude は前の回答からタスクの性質を読み取り、自明な項目は「ここは自明そうなので省略しますか?」とユーザーに確認した上でスキップしてよい (例: 「gem 追加するだけ」のタスクで実装方針 2-3 案は不要)。

1. **ゴール**: この Issue が close される時、何が動いている / 何が確認できる状態か
2. **実装方針の候補**: 2-3 案を Claude 側から提示、それぞれのトレードオフを 1 行ずつ。推奨を明示。ユーザーが選ぶ (自明なら省略可)
3. **完了条件**: テスト含めたチェックリスト (3-5 項目、自明なら省略可)

途中でユーザーが「もういい」「中止」と言ったら無理に進めない。Step 6 の (C) ルートへ。

### Step 5: Issue 本文を組み立てて preview

以下のテンプレで markdown を組み立て、コードブロック内でプレビュー表示する。

```markdown
## 背景

(Step 2 の内容)

## ゴール

(Step 4-1 の内容)

## 実装方針 (確定済み)

(Step 4-2 で選ばれた案。「なぜこの案を選んだか」も 1-2 行で書く)
※ 省略した場合: 「自明のため省略」と一行入れる

## 完了条件

- [ ] (Step 4-3 の項目 1)
- [ ] (項目 2)
- [ ] ...
- [ ] PR 作成 → 3 者並列レビュー → マージ

## 関連

- (関連 Issue / PR / メモがあれば)
- 親メモ: `memory/project_weight_dialy.md` (該当箇所があれば)
```

タイトルは Step 1 のトピックから 1 行で要約 (例: `feat: RSpec を導入してテスト基盤を整える`)。

### Step 6: ユーザー判断を仰ぐ (3 択)

「この内容で進めますか?」と聞き、3 つの選択肢を提示する:

#### (A) Issue 化する → 即起票

シェル展開リスクを避けるため、本文は **`--body-file` または heredoc 経由** で渡す:

```bash
gh issue create --title "<タイトル>" --body-file <(cat <<'EOF'
<本文>
EOF
)
```

実行して URL を返す。ユーザーが「今すぐ着手」と言ったら、続けて作業ブランチを切ってもよい (`git checkout -b feature/<内容>`)。

#### (B) まだ早い → Backlog Issue に積む

「Backlog Issue の扱い」セクションのロジックに従って checkbox 1 行を追記。
**追記完了後、Backlog Issue の URL と追記した行をユーザーに返す** (「本当に積まれた」ことを確認できるように):

> #XX に追加しました: `- [ ] <要約> (YYYY-MM-DD)`
> URL: https://github.com/.../issues/XX

#### (C) 中止 → 何もしない

「OK、また気が変わったら呼んでください」で終了。

## Backlog Issue の扱い

Backlog Issue は「未整理のアイデアを溜める常時オープン Issue」。
熟したら独立 Issue に昇格し、Backlog 側はチェックして横線を引く。

### 存在確認と自動作成 ((B) ルートのみ実行)

**(A) ルートでは実行しない**。独立 Issue を即起票するルートで Backlog 確認は不要。

(B) ルートに入った時、初回参照なら以下を実行:

**search クエリは日本語タイトル検索が当たらないことがあるため、全件取得 → title 完全一致フィルタが確実**:

```bash
gh issue list --state open --json number,title --limit 100 \
  | jq '[.[] | select(.title == "Backlog: 未整理のアイデア・思いつき")] | .[0].number // empty'
```

- 結果が空 → 存在しない → 新規作成 (下記)
- 結果が number 1 つ → そのまま使う
- 念のため: タイトル完全一致が複数件見つかった場合は、`gh issue list --state open --json number,title --limit 100 | jq '[.[] | select(.title == "...")] | length'` で件数を確認し、2 件以上ならユーザーに「Backlog Issue が複数あります。古い方を close してください」と通知

#### 存在しなければ新規作成

```bash
gh issue create --title "Backlog: 未整理のアイデア・思いつき" --body-file <(cat <<'EOF'
<本文>
EOF
)
```

本文テンプレ:

```markdown
このプロジェクトで「いつかやるかも」「思いついた」レベルのアイデアを溜める常時オープン Issue。

## 運用ルール

- 1 行 1 アイデア (markdown checkbox)
- 末尾に追加日 `(YYYY-MM-DD)` を付ける
- 熟したら独立 Issue に昇格、ここではチェックして取り消し線で消化を可視化
- close しない (常時オープン)

## アイデア

(最初のアイデアは `/plan-issue` コマンドで追加されます)
```

#### 存在すれば 1 行追記

シェル展開で本文の改行や引用符が壊れるのを避けるため、**`--body-file` 経由で更新**する:

```bash
# 現在の本文を取得 (変数経由でシェル展開を避ける)
current_body=$(gh issue view <num> --json body --jq .body)

# 末尾に 1 行追加 (heredoc で組み立て、--body-file <(...) で渡す)
gh issue edit <num> --body-file <(cat <<EOF
${current_body}
- [ ] <ユーザーの要約 (今日の対話を 1 行に圧縮)> ($(date +%Y-%m-%d))
EOF
)
```

追記する行のフォーマット: `- [ ] <ユーザーの要約> (YYYY-MM-DD)`

## 注意事項

- `gh issue create` / `gh issue edit` は外部に影響する。**必ずユーザーの明示的承認を得てから** 実行
- このコマンドは「対話による設計」が主目的。起票は副産物。質問を急がない
- $ARGUMENTS が空でも、ユーザーが対話の途中で「やっぱり要らない」と言ったら無理に Issue 化しない
- 1 回の対話で 1 つの Issue を起こすのが基本。サブタスクが派生したら「これは別 Issue として後で詰めますか?」と確認する
- 教材性: 対話の中で「なぜこの実装方針を選んだか」を残すのが weight_dialy の方針。後輩が読むことを意識した本文を組み立てる
- Backlog Issue の存在チェックは (B) ルートのみ。無駄な API 呼び出しを避ける
