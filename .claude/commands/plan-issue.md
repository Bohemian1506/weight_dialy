# /plan-issue

「1 Issue / 1 機能 / 1 PR」運用を支援するコマンド。
**対話は最小**。背景 + ゴールを 1 ターンで聞き、自明箇所はスキップする (= 9 日間死蔵を解除した軽量版、Issue #256)。

## 入力

トピック: $ARGUMENTS

ユーザーが軽くトピックだけ渡している場合は上に文字列が入る。空ならまずトピック自体を聞く。

## フロー

このコマンドは「設計を詰める対話」が主目的、起票は副産物。ただし **問いを盛らない**。

### Step 1: トピック確認

- $ARGUMENTS が空 → 「次に何をやりますか? 短い言葉で OK です」
- $ARGUMENTS にトピックがある → 「『$ARGUMENTS』を詰めますね」と確認してから次へ

### Step 2: 背景 + ゴールを 1 問で聞く (= 統合質問)

**1 ターンで両方拾う**:

> 「何が困っていて、解決後はどう動いている状態にしたい? (= 背景 + ゴールを 1 ターンで)」

ユーザーが片方しか答えなかった場合のみ、足りない方を 1 問だけ追問する (= 2 問目を機械的に出さない)。

### Step 3: 重複チェック (= keyword 抽出できた時のみ)

**毎回機械的に走らせない**。Step 2 の回答から以下のいずれかが抽出できた場合のみ実行:

- 既存機能名 (例: 「ログイン」「Webhook」「ダッシュボード」)
- 似た領域のキーワード (例: 「フォーム」「OAuth」「カード」)
- Issue 番号への明示的言及 (例: 「#143 と関連」)

抽出できる場合:

```bash
gh issue list --search "<抽出キーワード>" --state all --limit 5
```

似た Issue が見つかったら「#XX が近そうですが、別 Issue にしますか? 統合しますか?」と判断を仰ぐ。

抽出できない (= 完全新規領域) ならスキップ。

### Step 4: 実装方針 (= 複雑度高い時のみ提示)

**自明なら確認すら取らずスキップ**。Claude が頭の中で A 案/B 案を浮かべられる場合のみ提示:

- 2-3 案を Claude 側から提示
- 各案のトレードオフを 1 行ずつ
- 推奨を明示、ユーザーが選ぶ

1 本道 (= 案が 1 つしか浮かばない) ならこの Step を飛ばす。

### Step 5: 完了条件 (= 自明なら省略可)

3-5 項目のチェックリスト。自明なら「ここは自明そうなので省略しますか?」と確認した上でスキップしてよい。

### Step 6: Issue 本文を組み立てて preview

以下のテンプレで markdown を組み立て、コードブロック内でプレビュー表示:

```markdown
## 背景

(Step 2 の背景部分)

## ゴール

(Step 2 のゴール部分)

## 実装方針 (確定済み)

(Step 4 で選ばれた案。「なぜこの案を選んだか」も 1-2 行で書く)
※ Step 4 を飛ばした場合: 「自明のため省略」と一行入れる

## 完了条件

- [ ] (Step 5 の項目 1)
- [ ] (項目 2)
- [ ] ...
- [ ] PR 作成 → 3 者並列レビュー → マージ

## 関連

- (関連 Issue / PR / メモがあれば)
```

タイトルは Step 1 のトピックから 1 行で要約 (例: `feat: RSpec を導入してテスト基盤を整える`)。

### Step 7: ユーザー判断を仰ぐ (= 2 択)

「この内容で進めますか?」と聞き、2 つの選択肢を提示する:

#### (A) Issue 化する → 即起票

シェル展開リスクを避けるため、本文は **`--body-file` または heredoc 経由** で渡す:

```bash
gh issue create --title "<タイトル>" --body-file <(cat <<'EOF'
<本文>
EOF
)
```

実行して URL を返す。ユーザーが「今すぐ着手」と言ったら、続けて作業ブランチを切ってもよい (`git checkout -b feature/<内容>`)。

#### (C) 中止 → 何もしない

「OK、また気が変わったら呼んでください」で終了。

> **(B) Backlog ルートは Issue #256 で廃止**: Backlog Issue #8 が Day 1-7 で 11 件中 2 件のみ昇格 (= 18%)、Day 8-9 ゼロ更新で死蔵化。「軽量 Issue は 1 PR 完結」 ルール (= memory `feedback_light_issue_one_pr_completion.md`) 確立で積む必要が消失したため、(A) Issue 化 or (C) 中止 の 2 択に簡略化。

## 注意事項

- `gh issue create` は外部に影響する。**必ずユーザーの明示的承認を得てから** 実行
- このコマンドは「対話による設計」が主目的だが、**問いを盛らない**。Step 3-5 はそれぞれ条件付き発火
- $ARGUMENTS が空でも、ユーザーが対話の途中で「やっぱり要らない」と言ったら無理に Issue 化しない
- 1 回の対話で 1 つの Issue を起こすのが基本。サブタスクが派生したら「これは別 Issue として後で詰めますか?」と確認する
- 教材性: 対話の中で「なぜこの実装方針を選んだか」を残すのが weight_dialy の方針。後輩が読むことを意識した本文を組み立てる
