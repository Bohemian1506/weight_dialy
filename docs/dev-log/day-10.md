## How to apply (= Day 10 由来、適用済みの操作手順)

- **Skill / Command 自動化実装は論点 4 件 (= 例: 分割 / テンプレ / 発火 / 同期) を対話で詰めてから着手** (= dev-log-merge での A/B/C/D 4 件確定実例): 機械化対象が複雑な時、論点を局長と対話で詰めることで、後続実装で迷わず一気通貫に進める。3 者並列レビューで再相対化される論点も、対話で詰めた経緯が PR description に残るため格下げ説明が容易
- **ドッグフーディングは Skill マージ直後の同日に 1 周回す** (= 本 day-10.md 自身が初実演): Skill 完成日に dev-log 追記サイクルで Skill を実演することで、Issue 完了条件の最終項目を満たしつつ Skill の動作検証になる。Day 11 以降の発火 2 回目以降で観測される最適化は SKILL.md 補足追記候補として残す (= 「Claude 側に材料がある場合は下書き提示」 が Day 10 観測の 1 例目)
- **メタ相対化 (= 局長承諾を 3 者レビュー視点で再相対化) は新サブカテゴリとして許容**: memory `feedback_self_proposal_relativization.md` の過去 4 例は「自身の提唱を自身で発見して格下げ」 (= 自己完結) だが、本 Day 10 学び 36 候補は「**局長承諾事項を 3 者レビューで再相対化**」 (= メタ判断)。局長判断軸を覆す形の再相対化を許容する運用は、教材性 (= weight_dialy「Issue = 仮説 / PR = 確定、上書き可」 思想) と整合する
