## 🧠 教訓ハイライト (= 5 件確立)

### 教訓 1: Capacitor CLI 8.x の Node >= 22 要件 = `package.json` `engines` 事前確認

外部 CLI (= Capacitor CLI など) を CI で使う時、依存ライブラリのメジャー版要件を事前確認するワークフローが必要。`package.json` の `engines` フィールドに `"node": ">=22"` を明記しておけば、ローカル `npm install` 時に警告が出るため、CI ビルド前に気付ける。

**How to apply**: 新規外部 CLI 導入時は `package.json` engines を必ず明記。3 者並列レビューでも「依存ライブラリのメジャー版要件」 は盲点になりやすい。

**学び 36 サブカテゴリ候補**: 「コード内物理保存型」 4 つ目候補 = `engines` field を物理ガードに使う発想 (= memory `feedback_self_proposal_relativization.md` の不発閾値 3/5 例事前確定ルール上、4 つ目観測として記録)。

**※ カウント保留中**: memory `feedback_self_proposal_relativization.md` は Day 14 朝の strategic 指摘で「定義明文化先行ルール」 適用中 (= 「コード内物理保存型」 を 2 種類 = 墓標型 + 数式 WHY 型 で先行確定、その後 3 例目以降を逆引き分類)。本観測は **定義固定後に再判定** するため即カウント反映は保留。

### 教訓 2: XML コメント `--` 禁止仕様 = ローカル AAPT 警告 ≠ CI AAPT2 strict error

XML 1.0 §2.5 で XML コメント (`<!-- ... -->`) 内の double dash (`--`) は禁止仕様だが、Android Studio 内 AAPT は警告止まり、GitHub Actions の AGP 8.x + AAPT2 strict mode は error 化する。子 6 (= #125) で実機 / エミュレータビルドが通っていたファイルが CI で詰まる「環境差バグ」 の典型。

**How to apply**: `bin/setup` で `xmllint --noout` を pre-commit hook 化する検討は v1.1+ で。

### 教訓 3: メタ循環の罠 = `--` 禁止注記内に `"--"` 引用で再失敗

PR #308 で「XML コメント内 `--` 禁止」 を説明する補足注記を追加した際、「CSS の `"--"` プレフィックス」 と引用したつもりが、文字列リテラル内に依然として 2 連続ハイフン自体が含まれていて再失敗。

**How to apply**: memory `feedback_grep_zero_after_fix.md` (= fix PR は grep ヒット 0 件を Test plan に書く) を**コメント文字列自体にも適用** すべき。同 memory に「修正時は注記文自体も grep」 1 行追記候補。

### 教訓 4a: 自己完結型 6 例目候補 = Issue 推奨は永続契約でない

memory `feedback_self_proposal_relativization.md` の自己完結型カテゴリ 6 例目**候補**: Issue #145 本文「推奨 C (ウォームアップ) + 案 A (フォント同梱)」 を「**A 単独 + C 撤回**」 で PR #311 確定。Issue 推奨は v1.0 提出前判断 (= 案 C で当日運用)、v1.1 軽量 3 連完走後 + 閉会式言及前提に状況変化 → 案 A 前倒し + 案 C 撤回が筋良いと判断。

### 教訓 4b: Variable Font 検出手法 (= Google Fonts CSS API レスポンスの URL 重複)

**副次知見**: Google Fonts CSS API の Caveat レスポンスで 500 と 700 の latin subset URL が同一 → **Variable Font 検出**。@font-face で `font-weight: 500 700` range 指定により 1 ファイル 2 weight 賄える。Kalam は static font (= weight 別 URL)、Patrick Hand は 400 単独 = 各 family の variable / static 性質を CSS API 経由で検証する手法を確立。

**How to apply**: woff2 を直 DL する時、CSS API レスポンスの URL を `font-family + font-weight` の組合せでマトリクス化して重複検出。重複あり = variable font 候補 → @font-face の `font-weight: <min> <max>` で 1 ファイル多 weight 化、ファイルサイズ削減。

## How to apply

- **外的状況変化への撤退ライン 3 段設計**: 「閉会式直前 = 緊急タスク発生」 のような状況変化に対し、デッドライン逆算 + マスト/ベター/任意切り分け + 撤退ライン 3 段事前明示 で判断遅延ゼロ。本日 15:30 着手 → 17:00 1 段目クリア → 19:00 全完了 (= 50 分バッファ)

- **Issue 推奨は永続契約でない**: Issue 本文の「推奨 C + A」 が状況変化で「A 単独 + C 撤回」 に転回する = 自己完結型カテゴリの典型。Issue=仮説 / PR=確定 の枠組みを再確認
- **fix PR の grep ゼロ確認は注記文自体にも適用**: PR #308 のメタ循環の罠を防ぐため、修正時の grep には注記文も含める。memory `feedback_grep_zero_after_fix.md` 適用範囲拡張

- **自己完結型カテゴリと即日完走パターンの分離**: Issue 起票日 < PR マージ日 (= 1 セッション境界またぐ) を自己完結型の必須条件にする。同セッション完走は別カテゴリ「即日完走型」 として別途観測、定義固定先行
