## 🧠 主要な学び

### 教訓 1 候補 (= 1 例目、観測継続): CLAUDE.md 規範化と memory 化は **撤回コスト** で軸を分ける

3 者レビュー (= strategic) で発見した **仮説**。本 PR が 1 例目、確定は 2-3 例目観測後 (= memory `feedback_self_proposal_relativization.md` 3 回観測ルール準拠):

- 案 A (= CLAUDE.md 規範化) は `/insights` 1 例で実施したが、これは 3 回観測ルール (= memory 化判断) とは別軸の **仮説**。
- 理由: **撤回コストの非対称性**:
  - **memory 化** = MEMORY.md 更新 + 関連 PR 連鎖 → 撤回コスト高 → 3 回観測ルール厳格適用
  - **CLAUDE.md 規範化** = revert PR 1 本 → 撤回コスト低 → 摩擦の早期解消優先
- 候補事例: main 直 push 禁止ガードも 1 件の事故で即機械化された前例あり (= 仮説の 0 例目候補、観測軸が異なるので別事例として扱うか要検討)。

**How to apply**: 観測 2-3 例目 (= 別の CLAUDE.md 規範化判断) が出るまで「仮説」 として保持。本 PR の連番ルール更新自体が 2 例目候補になる可能性 (= 同日内なので慎重に判定)。

### 教訓 2 (= 観測ログ): `/insights` レポートは学び 36 (= 外部触媒型) の **累計 8 件目 (= Day 11-3 反証方向転換後の 1 例目)**

- 学び 36 累計事例: 確定 2 + 自爆 1 + 自走 3 + Production 1 + **/insights 1 (= 本セッション)** = 累計 8 件。
- Day 11-3 で **旧仮説反証方向転換** されており、本事例は **転換後の 1 例目**。
- 単独で気付くのは難しい摩擦 (= Claude が canon を確認せず仮定で進む) が「外部統計の上位指摘」 で言語化された (= 外部触媒型 = **外部からの観測・統計によって自プロジェクトの隠れた摩擦を可視化するパターン**)。

**How to apply**:
- `/insights` を **仕組み化セッション開始時** + **月 1 回の定期実行** + **Claude tooling ロードマップ第二群着手前** に確認する。
- 「Anthropic 全体統計の筆頭摩擦」 + 「自プロジェクトの memory 体系」 が衝突する箇所を仕組み化候補とする。

### 教訓 3: 段階導入 (A → B → C) の判断基準を **事前定量化**することで「やめどき」 を明文化

- 案 A 採用時に、案 B 起動条件を 3 軸 (= 自発発火率 N/M / 矛盾発見回数 / 抜け漏れ) で事前定量化。
- Day 14-2 教訓 4「Kotlin 越境 vs JS 完結の境界線越え禁止」 + Day 13 PR #294「不発閾値 3/5 例事前確定」 と同系統 = **撤退・進行判断の事前数値化**。
- PR #294 が「不発判定」 にフォーカスだったのに対し、本教訓は「進行判定」 = **対補完関係** (= 重複ではない)。
- 試運用後の判断者 (= 局長) の主観を減らし、後輩追跡可能にする。

**How to apply**: 段階導入を採用する PR では、次段階の起動条件を 3 軸以上で事前定量化して PR description / Issue 本文に記載する。

### 教訓 4 (= サブカテゴリ 4 候補「数式 WHY 型」 観測継続、ただし異種合算 n=3 は 2 サブパターン分割候補): i18n 化非対象の 1 行 WHY コメント物理保存

PR #346 (= home i18n 抽出) で **意図的に yml 化対象外** とした 4 項目に **1 行コメントで非対象理由を物理保存** した (= strategic-reviewer 論点 2 由来):

- `_dashboard.html.erb:68` `week_days` = 1 文字 / view 局所定数
- `_dashboard.html.erb:153` SVG `"kc"` = layout-tied truncated 単位ラベル
- `_dashboard.html.erb:199` AI `advice.*` = service 由来動的生成、SSOT は service 側
- `_dashboard.html.erb:219` `"余裕"` sentinel = AI service 由来 enum 役割、yml 化は fragile な結合化

これは memory `feedback_self_proposal_relativization.md` サブカテゴリ 4 候補 **「数式 WHY 型」** (= Day 14 PR #299 + #302 で 2 件観測済) の **3 例目候補** に該当。

**ただし 3 例観測ルールの本来想定 (= 同一型を 3 回独立観測) からは半歩はみ出る**: 1 件目 #299 + 2 件目 #302 = 「実装側 WHY」 (= 数式・実装意図の物理保存)、本件 #346 = 「除外側 WHY」 (= やらない理由の物理保存)。サブパターンとして分割すれば実質 **「実装 WHY 型 n=2 + 除外 WHY 型 n=1」** で n=3 未達と再カウントされる可能性が残る (= 本 PR 3 者並列レビュー strategic 論点 1 指摘)。

**How to apply**: 翌セッション冒頭で **n=3 認定の再 canon-check を必須前提** とし、(a) 同一型 3 回として確定 → memory 反映 PR 起票 / (b) 2 サブパターン分割 → 「除外 WHY 型 = 新サブカテゴリ候補」 観測モードで別 Issue 起票、の 2 分岐で判断する (= How to apply 11 で追跡、撤回コスト軸分離 1 例目観測の精神 = 撤回コスト低い dev-log 段階で表記弱化済)。

### 教訓 5 (= 規約進化 5 候補 2 例目観測): 装飾 class と UI 文言の分離

PR #337 (= about) で 1 例目観測 → PR #346 (= home AI カード `<span class="sketch-hl-yellow">動いた</span>`) で **2 例目観測達成**。3 例観測ルール (= memory `feedback_self_proposal_relativization.md`) で残 1 例 = 次画面 (= history / weight 入力等の i18n 抽出または別 view 編集時) で 3 例目発火条件成立すれば memory 化 GO。

**How to apply**: 後続画面の i18n 抽出 PR (= `feature_x.yml` 起こし時) で `<span class>` / `<br>` / `<i>` 等装飾要素を必ず view 側に残す + テキストを yml で複数 key に分割するパターンを意識的に観測。3 例目発火で memory `project_phase_1_1_refactoring.md` 規約進化 5 候補から「規約確定」 へ昇格、Issue #348 (= docs/ 化判断ログ) の含む項目に追加候補。
