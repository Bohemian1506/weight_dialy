## 🧠 学び (= 副次知見、Day 13+ 観測候補)

### 副次知見 1: カウンタベースの UX 制御 = 5xx 連続失敗特化での状態管理移行 (= strategic 論点 3 反映、言い切り格下げ)

> ぜんぶ忘れていい、この 1 シーンだけ。
> 「しばらく待って」 と書いても押す人は押す。**ボタンを消す** のが本当の「待つ」 誘導 (= 5xx 連続失敗の文脈で)。

Issue #273 完了時点では retry 文言の「しばらく待ってから」 で連打抑制を狙ったが、design-reviewer から「無限ループ感が残る」 指摘。文言ベースの抑制 → カウンタベースの状態管理に進化させた。

→ **5xx 連続失敗のような連打回避特化文脈では** UX 制御は文言だけで完結せず、状態 (= 試行回数) に応じて UI 要素自体を変える方が誘導が強い局面がある。「言葉で抑止」 vs 「物理的に押せない」 の対比は文脈次第:
- **状態管理が強い局面** = 連打が無害でない (= 連続 5xx = サーバー負荷増加、retry 連打 = 復旧妨害)
- **文言で足りる局面** = 連打が無害 or 1 回試行で結論が出る (= フォーム submit 後の confirm、削除確認等)

→ **本 PR の判定**: 5xx 連続失敗は前者 = 状態管理採用が正当。**他の連打抑制ロジックに転用するかは実装側の文脈次第** (= 一般化は危険、副局長 strategic 論点 3 由来の格下げ)。

### 副次知見 2: リセットタイミング 3 箇所の意味付け

`retryCount = 0` を 3 箇所に置く設計:

| タイミング | 意味 |
|---|---|
| sync 成功時 | 「成功した = サーバーは復旧」 = 次の 5xx はゼロからカウント |
| settings 遷移時 | 「token 再生成で文脈変更」 = 新規セッション扱い |
| connect 時 | 「ページ再表示 = セッション初期化」 = turbo navigation 跨ぎでも安全 |

→ **状態リセットは「文脈の変わり目」 で行う**。sync 成功・遷移・再 connect はそれぞれ「直前の失敗系列が無効化される瞬間」。1 リセット = 1 文脈変更の対応関係を保つことで、後輩が読んだ時にどこをリセット箇所として追加すべきか判断しやすい。

## How to apply

- **連打抑制は文脈次第で文言 or 状態管理**: 連打が無害な場面 (= フォーム submit / 削除確認) は文言で足りる、連打が無害でない場面 (= 連続 5xx でサーバー負荷増 / 復旧妨害) は状態管理が強い (= 一般化は危険、文脈判断必須)
- **状態リセットは文脈の変わり目で行う**: 1 リセット = 1 文脈変更 (= 成功 / 遷移 / 再 connect) の対応関係を保つ、後輩が追加箇所を判断しやすい
- **コメント寿命を 3 分類で扱う**: WHY 設計判断 (= 永続) / 時限的予告 (= Issue close で駆除) / TODO (= 修正完了で駆除)。本副次知見 3 の射程は時限的予告のみ、WHY は memory `feedback_code_as_communication.md` で永続が原則
- **grep ヒット 0 件確認は JS / view / Ruby 含む全体で実施**: 陳腐化コメント駆除では拡張子限定の grep が漏れ穴 (= 本 PR で view 側コメントを code-reviewer に発見されて反省)
- **数値ハードコードは「仮置き」 明示で予防接種**: `RETRY_THRESHOLD = 2` のような独立根拠が弱い数値は「仮置き、再評価候補」 コメントで自己提唱の言い切り回避 (= memory `feedback_self_proposal_relativization.md` 集計慣習の「不発閾値 3/5 仮置き明示」 と同パターン)
