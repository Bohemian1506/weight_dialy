## 🧠 学び (= 副次知見、Day 13+ 観測候補)

### 副次知見 1: カウンタベースの UX 制御 = 文言だけの連打抑制から状態管理への移行

> ぜんぶ忘れていい、この 1 シーンだけ。
> 「しばらく待って」 と書いても押す人は押す。**ボタンを消す** のが本当の「待つ」 誘導。

Issue #273 完了時点では retry 文言の「しばらく待ってから」 で連打抑制を狙ったが、design-reviewer から「無限ループ感が残る」 指摘。文言ベースの抑制 → カウンタベースの状態管理に進化させた。

→ UX 制御は文言だけで完結しない、状態 (= 試行回数) に応じて UI 要素自体を変える方が誘導が強い。**「言葉で抑止」 < 「物理的に押せない」** の対比は他の連打抑制ロジック (= フォーム submit / 重い処理ボタン等) にも転用可能。

### 副次知見 2: リセットタイミング 3 箇所の意味付け

`retryCount = 0` を 3 箇所に置く設計:

| タイミング | 意味 |
|---|---|
| sync 成功時 | 「成功した = サーバーは復旧」 = 次の 5xx はゼロからカウント |
| settings 遷移時 | 「token 再生成で文脈変更」 = 新規セッション扱い |
| connect 時 | 「ページ再表示 = セッション初期化」 = turbo navigation 跨ぎでも安全 |

→ **状態リセットは「文脈の変わり目」 で行う**。sync 成功・遷移・再 connect はそれぞれ「直前の失敗系列が無効化される瞬間」。1 リセット = 1 文脈変更の対応関係を保つことで、後輩が読んだ時にどこをリセット箇所として追加すべきか判断しやすい。

### 副次知見 3: 過去 PR の「別 Issue で対応」 コメント整理 = 完了時点で陳腐化を駆除

Issue #273 完了時点のコメント:

```js
// retry 文言の「しばらく待ってから」 は連打抑制を兼ねる (= 学び `feedback_one_change_two_bugs.md` 1 修正 2 不具合の前提通り、
// 5xx 連続失敗時のボタン非表示誘導は別 Issue #276 で対応、本 PR はカウンタ無し)。
```

本 PR (= Issue #276 完了) でこのコメントは陳腐化 → 削除。**「将来やる」 コメントは実装完了で削除する** = code-as-communication 原則 (= memory `feedback_code_as_communication.md`)。設計判断は PR description より fail-fast / 永続コメントで伝える、ただし**「別 Issue で対応」 系コメントは時限的で実装後はノイズ**。

## How to apply

- **連打抑制は文言より状態管理が強い**: 「しばらく待って」 と書くだけより、カウンタ + ボタン非表示で物理的に押せない方が誘導が確実
- **状態リセットは文脈の変わり目で行う**: 1 リセット = 1 文脈変更 (= 成功 / 遷移 / 再 connect) の対応関係を保つ、後輩が追加箇所を判断しやすい
- **「将来やる」 コメントは実装完了で削除**: code-as-communication 原則、時限的コメントは実装後にノイズ化するので駆除
