## 🧠 教訓ハイライト (= 1 件、本日午後確立)

### 学び 34: Service 切り出し時の unit spec 追加判断は 4 軸 × 2 タイミング — **判断の前提が変わったら判断を更新する**

PR #244 (= Issue #242、Tier 3 唯一の候補完走) の strategic-reviewer 提唱。午前学び 33 (= fat controller の Service 切り出し 4 軸) のうち **テスタビリティ軸** をさらに精緻化。

**学び 33 のサブ項目ではなく独立学びとして分離した理由**: タイミング軸 (= **いつ追加するか**) が、学び 33 の 4 軸 (= **何を切り出すか / 何を残すか**) とは別の問いだから。前者は時間軸、後者は構造軸 — 道具選定シリーズの **1 軸の縦深堀り** という新パターン (= 横展開 4 例目 = 学び 33 で完成 / 縦深堀り 1 例目 = 学び 34)。

> **用語注記 — 縦深堀り**: 横展開 (= 学び 27/30/32/33 のように **新しい道具選定** を並べる) に対して、**既存の 1 軸をさらに掘り下げる** こと (= 学び 33 のテスタビリティ軸を本学び 34 の「4 軸 × 2 タイミング」 マトリクスに精緻化)。シリーズの拡張軸が 2 種類 (= 横 / 縦) あることで、Day 10 以降の学びは「**5 例目** (= 横展開) or **2 例目** (= 縦深堀り)」 のどちらでも記述可能。

#### 4 軸 × 2 タイミングの判定マトリクス

Service 切り出し時に unit spec を **いつ追加するか** を、4 軸 × 2 タイミングで考える。

| 軸 (= 学び 33 由来 + 拡張) | 切り出しと同時 | 切り出し後 follow-up |
|---|---|---|
| ① テスタビリティ (= request spec で十分か) | 別経路追加が確定済 → 追加 | 後から判明 → follow-up Issue で追加 |
| ② 契約のドキュメント化 | 切り出し時に契約が複雑なら → 同時 | 「読みやすさ向上」 動機で → follow-up |
| ③ 一貫性 (= 他 service との揃え) | プロジェクト慣習が確立済 → 同時 | 慣習に気づいた時 → follow-up |
| ④ 別経路追加 | 確定 → 同時 | 将来の備え → 動機としては最弱 |

**動機軸の強弱ルール**: ④ は **見込み** の動機 (= まだ起きてないことへの備え) で最弱、② ③ は **観測済み** の動機 (= 今読みにくい / 既存慣習にズレている) で強い。前者は保留で十分、後者は今やる根拠になる。

#### 本 PR (= #241 → #242 → #244) の実例

| タイミング | 判断 | 動機軸 |
|---|---|---|
| PR #241 (= 切り出し時) | unit spec 追加せず | ④ 別経路追加なし、request spec 十分 |
| Issue #242 (= 切り出し直後の起票) | 「将来別経路追加時に検討」 と保留 | ④ 主動機 |
| PR #244 (= 同日 follow-up 着地) | 前倒し追加 | ② 契約のドキュメント化 + ③ 一貫性 |

→ **動機軸が ④ から ② + ③ に変わった瞬間に、判断を更新して追加した**。

#### 「判断の前提が変わったら判断を更新する」 (= weight_dialy 最大の教材性)

本 PR の核心: PR #241 マージ時点では「unit spec 不要」 と service 冒頭コメントに明記していた → PR #244 で追加した時、**コメントを「不要な判断」 のまま残すと将来読者が混乱する** → **コメントを「判断履歴」 に書き換え** (= PR #241 時点 / PR #244 時点の両方を時系列で記録)。

これは memory `feedback_issue_as_decision_log.md` (= 「Issue 本文 = 判断ログ」) のコード版実装。**判断は時間とともに変わる**、その経緯ごと残すのが weight_dialy の判断ログ運用。

#### 学び 33 → 34 の進化軌跡 (= 道具選定シリーズの新パターン)

午前 (day-9-1) 末の学び 33 末尾「unit spec を機械的に追加しない / 必要になったら追加: Issue #242 候補」 → 本学び 34 で「② 契約のドキュメント化 / ③ 一貫性 動機なら前倒し」 に **判断更新**。シリーズが横展開 (= 27/30/32/33) だけでなく **縦深堀り** (= 33 → 34) も可能と実証。

How to apply:
- Service 切り出し時に unit spec を追加するか迷ったら **4 軸 × 2 タイミング** マトリクスで判定
- 動機軸が ④ (= 別経路追加、見込み) のみなら **保留 + Issue 起票** が筋 (= 学び 33 の testability 軸の保守的解釈)
- 動機軸が ② (= 契約のドキュメント化) や ③ (= 一貫性、観測済み) なら **前倒し追加** が筋 (= 軽量 Issue 1 PR で follow-up が望ましい、Day 9-2 で実証)
- **コードコメントに「保留判断」 を書いた後で前倒し実装するときは、コメントを「判断履歴」 に書き換える** (= 嘘コメント防止、weight_dialy 教材性の最重要原則)
- Day 9 道具選定シリーズと同様に「**使ってみたい衝動を構造軸で機械的に検証**」 の枠で考える (= 軸の精緻化が学び 33 → 34 の進化軌跡)
- Issue 起票自体が「保留判断の記録」 として機能する (= memory `feedback_issue_as_decision_log.md` 整合、判断ログ運用の二重実装)
