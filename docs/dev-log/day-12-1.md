### 副次知見 1: 事故対応の確度別 Issue 3 分割パターン (= strategic-reviewer 提案で明示的評価された運用)

> ぜんぶ忘れていい、この 1 シーンだけ。
> **OOM のような事故対応で複数の根本原因が見つかった時、「警告系 (確度高)」 / 「棚卸し調査系 (確度中)」 / 「仕様調査系 (確度低)」 の 3 種類に分けて、確度高をスピード PR で先行マージする** = 「全部まとめて 1 PR にすると確度の低い側 (= 調査が必要な側) に引きずられて確度高のマージが遅れる」 ことを防ぐ。

5/10 OOM クラッシュ対応で 3 つの根本原因が見つかった:

| 確度 | Issue | 種別 | 性質 | 結果 |
|---|---|---|---|---|
| **高** | #280 | 警告系 (= Puma 推奨設定への切替) | 仮説不要、設定 1 行変更 | **同セッション内で PR #283 即マージ** |
| **中** | #281 → #284 | 棚卸し調査系 (= SolidQueue Job 棚卸し) | 実態調査が必要、結果次第で方針確定 | 調査完走 + フォローアップ #284 起票 |
| **低** | #282 | 仕様調査系 (= Anthropic API 400 原因不明) | 仮説検証が必要、外部 API 仕様確認込み | 未着手、Day 12+ |

#### 1 PR にまとめると何が起きるか (= strategic-reviewer 論点 1 から)

「3 件まとめて 1 PR にする」 案を仮想シミュレーションすると:
- 確度高 #280 (= 設定 1 行) は確度低 #282 (= 仕様調査) の調査終了を待つ
- #282 の調査が長引けば、その間 OOM 再発リスクが残り続ける
- レビューも「Puma 設定の妥当性」 + 「SolidQueue 棚卸し方針」 + 「Anthropic API 仕様」 の 3 ジャンルが混在 = レビュアーの認知負荷大
- 1 PR の test plan が「Render Metrics 観測」 + 「Job 機能影響確認」 + 「AI 動作確認」 の 3 軸 = 1 セッションで完結困難

→ **3 分割は責務分離 + スピード両立の合理解**。本パターンが **n=1 の観測**、Day 12+ の事故対応で再現性確認候補 (= 2 件目で仮説強化、3 件目で確定 → memory 化判定)。

### 副次知見 2: SolidQueue 完全空運転発見 = MVP 後の機能棚卸し重要性 (= Issue #281 調査結論)

> ぜんぶ忘れていい、この 1 シーンだけ。
> **Rails 8 のデフォルト機能 (= Solid Trifecta、SolidQueue 含む) を「とりあえず入れた」 まま使わず放置すると、本番で不要なプロセスを常駐起動して 512MB を圧迫することがある** = MVP 後の機能棚卸しで「使っていない機能」 を能動的に発見する運用が必要。

Issue #281 の調査ステップ通りに棚卸しした結果:

| 棚卸し項目 | 結果 |
|---|---|
| `app/jobs/` の独自 Job クラス | **ゼロ** (= `application_job.rb` のみ、Rails 8 のデフォルト) |
| `perform_later` / `perform_now` 呼び出し | **0 件** (= アプリ全体で grep) |
| `deliver_later` (ActionMailer) 呼び出し | **0 件** |
| `recurring.yml` の job | **1 件のみ = `clear_solid_queue_finished_jobs` (= SolidQueue 自身の job table を掃除する自己掃除 job)** |
| `queue_adapter` | `:solid_queue` (= `production.rb:53` で明示) |

= **SolidQueue は「掃除すべき空テーブルを掃除する」 ためだけに 4 プロセス (Supervisor / Worker / Dispatcher / Scheduler) を常駐起動していた** = 完全な過剰起動。

#### 棚卸しの定型手順 (= grep 3 連発で 1 分以内に判定可能)

```bash
# 1. Job クラス棚卸し
ls app/jobs/

# 2. enqueue 呼び出し棚卸し
grep -rn "perform_later\|perform_now\|deliver_later" app/

# 3. recurring schedule 確認
cat config/recurring.yml
```

3 つすべてが「ゼロ or 自己掃除のみ」 なら、その機能は実質未使用 = 停止候補。Rails 8 デフォルトを「とりあえず入れた」 後の MVP 完成タイミングで一度棚卸しすると、不要常駐プロセスを発見できる。

### 副次知見 3: サイレント機能後退の検出パターン (= Issue #282 起票の背景、fallback 万能の罠)

> ぜんぶ忘れていい、この 1 シーンだけ。
> **「fallback が効いているからユーザー影響ゼロ」 の機能後退は、ログ精読の機会がないと永遠に気付かれない** = fallback 設計時に「fallback 発火回数の監視」 か「定期的な動作確認」 のどちらかが必要。

Anthropic API 400 BadRequest は **5/10 09:12 OOM 調査時の Render ログ精読** で初めて発覚:

```
[CalorieAdviceService] AI failed (
  Anthropic::Errors::BadRequestError:
  {url: "https://api.anthropic.com/v1/messages", status: 400, ...),
falling back to static
```

= 23:50 / 23:58 / 00:08 / 00:11 / 00:12 / 00:14 すべて発生 = **恒常的に失敗 → static fallback → ユーザーに見える挙動は変わらない**。

| 観点 | 状態 |
|---|---|
| ユーザー影響 | **ゼロ** (= static fallback で食品換算は表示される) |
| エラー監視 | **検知ゼロ** (= log error は出ているが alert 設定なし) |
| Day 4 動作確認時 | 動いていた (= 当時は AI 経路が成功) |
| 本来の機能 | **止まったまま** (= 何日間止まっていたか不明) |

#### サイレント機能後退の発生条件

1. **fallback 設計が「ユーザー影響ゼロ」 を達成している** = サービス継続性の観点では理想形
2. **でも本来の機能 (= AI 動的生成) は止まっている** = 機能後退
3. **ログには error が出ているが、誰も見ない** = 通知設計がない
4. → 本来 1 日で気付くべき機能後退が「OOM 調査の副次発見」 として N 日後に判明

#### 対策候補 (= Issue #282 でフォローアップ)

- 短期: ログから error 検出 → Slack / メール通知
- 中期: spec で本番 API 経路を週次叩く (= flaky リスクあり)
- 長期: AI 経路の成功率メトリクス可視化 (= Render Metrics でカウント)

本副次知見は **「fallback は神器」 の安心感がサイレント後退リスクを生む** という設計トレードオフの教材性として価値あり。

## How to apply (= Day 12-1 由来、適用済みの操作手順)

- **事故対応の確度別 Issue 3 分割パターン** (= 副次知見 1): OOM 等の事故で複数根本原因が見つかったら **「警告系 (確度高、設定変更で完結) / 棚卸し調査系 (確度中、調査結果次第で方針) / 仕様調査系 (確度低、仮説検証必要)」** に分けて、確度高をスピード PR で先行マージ。1 PR にまとめると確度低に引きずられて確度高マージが遅れるリスクを回避。本パターン Day 12+ の事故対応で再現性確認候補
- **MVP 後機能棚卸しの定型手順 (= grep 3 連発で 1 分判定)** (= 副次知見 2): `ls app/jobs/` + `grep -rn "perform_later\|perform_now\|deliver_later" app/` + `cat config/recurring.yml` の 3 つで Solid Trifecta の queue 機能利用実態が即判定可能。「ゼロ or 自己掃除のみ」 なら停止候補。Rails 8 デフォルト機能を「とりあえず入れた」 後の MVP 完成タイミングで実施推奨
- **サイレント機能後退の検出は「ログ精読の機会」 を作るしかない** (= 副次知見 3): fallback 設計済みの機能は **error log が出ても alert がなければ気付かれない**。対策は (1) ログ → 通知 / (2) 定期動作確認 spec / (3) 成功率メトリクス可視化 のいずれか。fallback 設計時にセットで決めるのが理想 (= 後付けは忘れる)
- **plan-issue Skill 連続起動で複数 Issue を順次対話で詰める** (= 局長指示「全ての作業を Issue にしましょう」 を受けた本セッション初実演): 3 件以上の Issue を起票する時、plan-issue Skill を 1 件ずつ起動 → 対話で詰める → 起票 → 次の Issue へ、を繰り返す運用が成立。Skill 自体は 1 セッション 1 件の対話設計だが、連続起動で複数 Issue 対応可能
- **Issue 起票前に外部仕様の前提を api-researcher で裏取り** (= Day 12-1 で誤った Issue 起票を回避した実例): 「Render Background Worker に分離」 という前提が誤り (= Free plan では使えない $7/月〜) と判明 → Issue 方針を「軽量化検討」 から「現状把握から始める調査 Issue」 に修正。**Issue ドラフト前の 1 ターン api-researcher 起動が ROI 高い**
- **設定 1 行 PR でも 3 者並列レビューを通す** (= 軽量 Issue 1 PR ルール 13 例目): PR #283 は render.yaml 1 ファイル + 1 行変更だが、3 者レビューで design-reviewer から 9 件の情報設計指摘 = コメント全面書き直しに繋がり教材性が大幅向上。**「軽微 PR でも 3 者レビュー必須」 ルールの実利性再確認** (= memory `feedback_always_three_reviewers.md` 実演)
