### 副次知見 1: 事故対応の確度別 Issue 3 分割パターン (= strategic-reviewer 提案で明示的評価された運用)

> ぜんぶ忘れていい、この 1 シーンだけ。
> **OOM のような事故対応で複数の根本原因が見つかった時、「警告系 (確度高)」 / 「棚卸し調査系 (確度中)」 / 「仕様調査系 (確度低)」 の 3 種類に分けて、確度高をスピード PR で先行マージする** = 「全部まとめて 1 PR にすると確度の低い側 (= 調査が必要な側) に引きずられて確度高のマージが遅れる」 ことを防ぐ。

5/10 OOM クラッシュ対応で 3 つの根本原因が見つかった:

| 確度 | Issue | 種別 | 性質 | 結果 |
|---|---|---|---|---|
| **高** | #280 | 警告系 (= Puma 推奨設定への切替) | 仮説不要、設定 1 行変更 | **同セッション内で PR #283 即マージ** |
| **中** | #281 → #284 | 棚卸し調査系 (= SolidQueue Job 棚卸し) | 実態調査が必要、結果次第で方針確定 | 調査完走 + フォローアップ #284 起票 |
| **低** | #282 | 仕様調査系 (= Anthropic API 400 原因不明) | 仮説検証が必要、外部 API 仕様確認込み | 未着手、Day 12+ |

#### 1 PR にまとめると何が起きるか (= strategic-reviewer 論点 1 から)

「3 件まとめて 1 PR にする」 案を仮想シミュレーションすると:
- 確度高 #280 (= 設定 1 行) は確度低 #282 (= 仕様調査) の調査終了を待つ
- #282 の調査が長引けば、その間 OOM 再発リスクが残り続ける
- レビューも「Puma 設定の妥当性」 + 「SolidQueue 棚卸し方針」 + 「Anthropic API 仕様」 の 3 ジャンルが混在 = レビュアーの認知負荷大
- 1 PR の test plan が「Render Metrics 観測」 + 「Job 機能影響確認」 + 「AI 動作確認」 の 3 軸 = 1 セッションで完結困難

→ **3 分割は責務分離 + スピード両立の合理解**。本パターンが **n=1 の観測**、Day 12+ の事故対応で再現性確認候補 (= 2 件目で仮説強化、3 件目で確定 → memory 化判定)。

### 副次知見 2: SolidQueue 完全空運転発見 = MVP 後の機能棚卸し重要性 (= Issue #281 調査結論)

> ぜんぶ忘れていい、この 1 シーンだけ。
> **Rails 8 のデフォルト機能 (= Solid Trifecta、SolidQueue 含む) を「とりあえず入れた」 まま使わず放置すると、本番で不要なプロセスを常駐起動して 512MB を圧迫することがある** = MVP 後の機能棚卸しで「使っていない機能」 を能動的に発見する運用が必要。

Issue #281 の調査ステップ通りに棚卸しした結果:

| 棚卸し項目 | 結果 |
|---|---|
| `app/jobs/` の独自 Job クラス | **ゼロ** (= `application_job.rb` のみ、Rails 8 のデフォルト) |
| `perform_later` / `perform_now` 呼び出し | **0 件** (= アプリ全体で grep) |
| `deliver_later` (ActionMailer) 呼び出し | **0 件** |
| `recurring.yml` の job | **1 件のみ = `clear_solid_queue_finished_jobs` (= SolidQueue 自身の job table を掃除する自己掃除 job)** |
| `queue_adapter` | `:solid_queue` (= `production.rb:53` で明示) |

= **SolidQueue は「掃除すべき空テーブルを掃除する」 ためだけに 4 プロセス (Supervisor / Worker / Dispatcher / Scheduler) を常駐起動していた** = 完全な過剰起動。

#### 棚卸しの定型手順 (= grep 3 連発で 1 分以内に判定可能)

```bash
# 1. Job クラス棚卸し
ls app/jobs/

# 2. enqueue 呼び出し棚卸し
grep -rn "perform_later\|perform_now\|deliver_later" app/

# 3. recurring schedule 確認
cat config/recurring.yml
```

3 つすべてが「ゼロ or 自己掃除のみ」 なら、その機能は実質未使用 = 停止候補。Rails 8 デフォルトを「とりあえず入れた」 後の MVP 完成タイミングで一度棚卸しすると、不要常駐プロセスを発見できる。

### 副次知見 3: サイレント機能後退の検出パターン (= Issue #282 起票の背景、fallback 万能の罠)

> ぜんぶ忘れていい、この 1 シーンだけ。
> **「fallback が効いているからユーザー影響ゼロ」 の機能後退は、ログ精読の機会がないと永遠に気付かれない** = fallback 設計時に「fallback 発火回数の監視」 か「定期的な動作確認」 のどちらかが必要。

Anthropic API 400 BadRequest は **5/10 09:12 OOM 調査時の Render ログ精読** で初めて発覚:

```
[CalorieAdviceService] AI failed (
  Anthropic::Errors::BadRequestError:
  {url: "https://api.anthropic.com/v1/messages", status: 400, ...),
falling back to static
```

= 23:50 / 23:58 / 00:08 / 00:11 / 00:12 / 00:14 すべて発生 = **恒常的に失敗 → static fallback → ユーザーに見える挙動は変わらない**。

| 観点 | 状態 |
|---|---|
| ユーザー影響 | **ゼロ** (= static fallback で食品換算は表示される) |
| エラー監視 | **検知ゼロ** (= log error は出ているが alert 設定なし) |
| Day 4 動作確認時 | 動いていた (= 当時は AI 経路が成功) |
| 本来の機能 | **止まったまま** (= 何日間止まっていたか不明) |

#### サイレント機能後退の発生条件

1. **fallback 設計が「ユーザー影響ゼロ」 を達成している** = サービス継続性の観点では理想形
2. **でも本来の機能 (= AI 動的生成) は止まっている** = 機能後退
3. **ログには error が出ているが、誰も見ない** = 通知設計がない
4. → 本来 1 日で気付くべき機能後退が「OOM 調査の副次発見」 として N 日後に判明

#### 対策候補 (= Issue #282 でフォローアップ)

- 短期: ログから error 検出 → Slack / メール通知
- 中期: spec で本番 API 経路を週次叩く (= flaky リスクあり)
- 長期: AI 経路の成功率メトリクス可視化 (= Render Metrics でカウント)

本副次知見は **「fallback は神器」 の安心感がサイレント後退リスクを生む** という設計トレードオフの教材性として価値あり。

### 副次知見 4: 3 者並列レビューの冗長性が Blocker 救済として実利を産んだ初観測 (= memory `feedback_always_three_reviewers.md` の実利性確認)

> ぜんぶ忘れていい、この 1 シーンだけ。
> **「軽微 PR でも 3 者並列レビューを通す」 ルールは「教材性のため」 と理解されがちだが、実は「自分が見落とすバグを他人が見つける」 という実利性こそが核心** = PR #286 で初めて実演された。

PR #286 (= SOLID_QUEUE_IN_PUMA=false 変更) は当初「ENV 1 行変更だけ」 の超軽量 PR と想定していたが、3 者並列レビューで:

- **strategic-reviewer**: 同 Blocker を **devcontainer で実検証** (= `TRUTHY-plugin-loads`) して独立に指摘

= **両 Agent が同じ Blocker を独立に発見** = レビュー冗長性が初めて実利を産んだ瞬間。

#### 3 者レビューを通さなかった場合のシミュレーション

- PR #286 を「ENV 1 行変更だけ」 でマージ → 本番デプロイ → Render ログに `SolidQueue Started Supervisor/Worker/Dispatcher/Scheduler` が出続ける

= **3 者レビュー冗長性 (= 同じ Blocker を複数 Agent が独立に出す) が高 ROI 救済として機能**。

#### memory `feedback_always_three_reviewers.md` の進化

#### 副次的副次知見: 「Rails 標準パターンの落とし穴」 = `if ENV["X"]` 罠

`config/puma.rb` の Rails デフォルト記述 `plugin :solid_queue if ENV["SOLID_QUEUE_IN_PUMA"]` は **ENV 存在 = ON** という意味で、値による OFF を想定していない。**ユーザーが `"false"` を設定しても plugin はロードされる** という直感に反する挙動。memory 化候補 `feedback_env_truthiness_pitfall.md` (= 別 Issue 起票候補)。

### 副次知見 5: 「相関関係 ≠ 因果関係」 の罠 + Anthropic API クレジット枯渇真因 (= Issue #282 close)

> ぜんぶ忘れていい、この 1 シーンだけ。
> **デプロイ直後に何かが復活した時、デプロイと真因が独立した可能性を必ず検証する** = 「PR #286 を出したら AI が動き出した」 を見て「SolidQueue 停止が AI を救った」 と推測したが、真因はクレジット補充というアプリ外の出来事だった。

Issue #282 調査の最終フェーズで以下の時系列を観測:

| 時刻 (UTC) | イベント |
|---|---|
| 00:48-01:38 | 旧インスタンス (`n4sjm` / `65zbv` / `8trsh`) で `AI failed` 連発 |
| 01:46 | PR #286 (= SolidQueue 停止) デプロイ完了 → 新インスタンス `94zjn` |
| **01:45-01:48** | **新インスタンス `94zjn` で `AI suggestion ok` 成功!** |

→ 「SolidQueue 4 プロセス停止 → メモリ余裕 → Anthropic API 通信成功 → AI 復活」 という誤った因果関係を推測。**真因は Anthropic API キーのクレジットがゼロになっていた** こと (= ユーザーが別作業中に発見、補充タイミングが PR #286 デプロイと偶然近接)。

#### 教訓 4 つ

1. **時間的近接だけで因果関係を結論付けてはいけない** (= 観測点 n=1 で複数候補が同時発生した場合、独立検証なしでは特定できない)
2. **Anthropic API は credit 枯渇時に `400 invalid_request_error` を返す** (= 401 Unauthorized でなく 400、エラータイプも誤誘導的、副次知見 3 と同根の「サイレント機能後退」)
3. **エラーログの body 切り捨てが真因発見を遅らせた** (= Issue #288 で `truncate(120)` → `truncate(2000)` に修正、本セッション完走、副次知見 6 と接続)
4. **fallback 万能の罠** (= 副次知見 3 と同根、API クレジット枯渇でも static fallback でユーザー影響ゼロのまま気付かれず継続、Issue #289 でクレジット監視 B + C 案採用)

#### 「次回 OOM 等の事故対応では『独立検証必須』 をチェックリスト化」 という運用学び

本副次知見は memory 化候補 (= `feedback_correlation_vs_causation.md`、別 Issue 起票候補)。observation n=1 + 時間的近接 = 仮説 1 件、複数候補が同時発生したら **独立検証で別個の真因可能性をつぶす** ことを習慣化。

### 副次知見 6: memory `feedback_grep_zero_after_fix.md` の polish PR 適用拡張 = 1 ファイル想定 → 同パターン全駆除 (= PR #290 で実演)

> ぜんぶ忘れていい、この 1 シーンだけ。
> **同じバグ修正パターンは 1 ファイルだけでなく、コードベース全体を grep して同パターンを全部駆除する** = bug fix 想定で確立した memory ルールを polish PR にも本能的に適用すべき、ただし「同パターン」 の判定軸は機械的でなく性質ベース。

| `feedback_grep_zero_after_fix.md` (= 同名パターン全駆除、bug fix 想定) | 本 PR で全件修正 (B 案) |

→ **B 案採用で衝突解消**。判断軸:
- 修正内容は完全に同じ (= `truncate(120)` → `truncate(2000)` の 5 文字置換 × 3 箇所)、別 Issue + 別 PR + 別 3 者レビューの ROI が低い
- 本 Issue の本質 (= 真因即発見の運用改善) は CalorieAdviceService 固有でなく weight_dialy 全 rescue 節に共通
- 軽量 Issue 1 PR ルールの「軽量」 は本 PR 内の追加修正に適用、本 PR は依然として軽量 (= +26/-3 行)

#### 性質判定で機械的駆除を回避 (= L24 除外の判断)

webhooks_controller.rb 3 箇所のうち **L24 (= `WebhookHealthDataIngestService::InvalidPayload`) は意図的に除外**:

| 行 | 経路 | 用途 | 判定 |
|---|---|---|---|
| L20 | `JSON::ParserError` | `Rails.logger.warn` (= ログ用) | `truncate(2000)` 修正 ✅ |
| L24 | `InvalidPayload` | **DB の `WebhookDelivery#error_message` column 保存用** | 意図的残置 + コメント根拠明示 |
| L29 | `ActiveRecord::RecordInvalid` | `Rails.logger.warn` (= ログ用) | `truncate(2000)` 修正 ✅ |

→ **「同パターン」 を文字列マッチでなく用途で判定** (= ログ用 vs DB 保存用)。strategic-reviewer の grep 指摘を鵜呑みにせず、性質判定で機械的除外を回避するのが正解。ログ用 `truncate(120)` は **grep 0 件達成** (= memory 要件満たす)。

#### 教訓

- **「polish PR でも同名パターン grep を本能的に通す」 のが正解** = 当初想定 1 ファイルが拡張されても、ROI 観点で本 PR 内吸収が筋

- memory `feedback_grep_zero_after_fix.md` の **polish PR 適用拡張** は別 Issue 起票候補 (= 本 PR スコープ外、Day 12+ で memory 更新 Issue 起票)

## How to apply (= Day 12-1 由来、適用済みの操作手順)

- **事故対応の確度別 Issue 3 分割パターン** (= 副次知見 1): OOM 等の事故で複数根本原因が見つかったら **「警告系 (確度高、設定変更で完結) / 棚卸し調査系 (確度中、調査結果次第で方針) / 仕様調査系 (確度低、仮説検証必要)」** に分けて、確度高をスピード PR で先行マージ。1 PR にまとめると確度低に引きずられて確度高マージが遅れるリスクを回避。本パターン Day 12+ の事故対応で再現性確認候補
- **MVP 後機能棚卸しの定型手順 (= grep 3 連発で 1 分判定)** (= 副次知見 2): `ls app/jobs/` + `grep -rn "perform_later\|perform_now\|deliver_later" app/` + `cat config/recurring.yml` の 3 つで Solid Trifecta の queue 機能利用実態が即判定可能。「ゼロ or 自己掃除のみ」 なら停止候補。Rails 8 デフォルト機能を「とりあえず入れた」 後の MVP 完成タイミングで実施推奨
- **サイレント機能後退の検出は「ログ精読の機会」 を作るしかない** (= 副次知見 3): fallback 設計済みの機能は **error log が出ても alert がなければ気付かれない**。対策は (1) ログ → 通知 / (2) 定期動作確認 spec / (3) 成功率メトリクス可視化 のいずれか。fallback 設計時にセットで決めるのが理想 (= 後付けは忘れる)
- **plan-issue Skill 連続起動で複数 Issue を順次対話で詰める** (= 局長指示「全ての作業を Issue にしましょう」 を受けた本セッション初実演): 3 件以上の Issue を起票する時、plan-issue Skill を 1 件ずつ起動 → 対話で詰める → 起票 → 次の Issue へ、を繰り返す運用が成立。Skill 自体は 1 セッション 1 件の対話設計だが、連続起動で複数 Issue 対応可能
- **Issue 起票前に外部仕様の前提を api-researcher で裏取り** (= Day 12-1 で誤った Issue 起票を回避した実例): 「Render Background Worker に分離」 という前提が誤り (= Free plan では使えない $7/月〜) と判明 → Issue 方針を「軽量化検討」 から「現状把握から始める調査 Issue」 に修正。**Issue ドラフト前の 1 ターン api-researcher 起動が ROI 高い**

- **3 者並列レビューの冗長性が Blocker 救済として実利を産む** (= 副次知見 4): code-reviewer + strategic-reviewer の独立判断が同じ Blocker を独立指摘した時こそ、軽微 PR でも 3 者並列レビューを通す価値が最大化される。「自分が書いた」 と思っているコードでも Rails 標準パターン (= `if ENV["X"]`) の罠は気付けない、外部視点の冗長性が必須。PR #286 で `plugin :solid_queue if ENV["SOLID_QUEUE_IN_PUMA"]` の truthy 罠を 3 者レビュー前に気付かなかった = 「自分が見落とすバグを他人が見つける」 仕組みが初めて Blocker 救済として機能
- **「相関関係 ≠ 因果関係」 = 時間的近接で結論付けない** (= 副次知見 5): デプロイ直後に何かが復活した時、デプロイと真因が独立した可能性を必ず検証する。本セッションは「クレジット補充タイミングが PR #286 デプロイと偶然近接」 で誤誘導を生んだ実例。次回 OOM 等の事故対応では「観測 n=1 で複数候補が同時発生したら独立検証必須」 をチェックリスト化。Anthropic API クレジット枯渇は `400 invalid_request_error` を返す (= 401 でなく) 命名誤誘導も同時に学び
- **polish PR でも同名パターン grep を本能的に通す** (= 副次知見 6): memory `feedback_grep_zero_after_fix.md` を bug fix だけでなく polish にも適用、strategic-reviewer の grep 指摘を待たず Claude 本体側で先回り grep する習慣。判定軸は機械的でなく性質ベース (= ログ vs DB 用 等の用途分類)。PR #290 で webhooks_controller.rb 3 箇所のうち L24 (= DB 用) は意図的除外 + コメント根拠明示で全駆除と性質判定を両立
- **plan-issue Skill 連続起動 + 親子分離 Issue パターン** (= 本セッションで 2 例実演): 調査 Issue (= #281 / #289) → 結果コメント投稿 + フォローアップ実装 Issue 起票 (= #284 / #291) + 親 Issue close、を 1 セッション内で完結する運用パターンが確立。Issue は仮説 (= 親調査) と確定 (= 子実装) で分離、調査結論次第で実装方針が変わるため別 Issue 切り出しが筋。本セッションで 2 件連続実演 = 「親調査 → 子実装」 が運用パターン化
