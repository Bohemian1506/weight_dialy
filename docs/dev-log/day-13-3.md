## 🧠 教訓ハイライト

**副次知見 3 件 10 秒サマリ**:
1. 後輩教材性は「立ち読み 10 秒 + 精読」 二層化 = H2 直下サマリ (= 10 秒把握) + 詳細セクション (= 精読) で動線分離
2. 太字密度の上限ルール = 各段落 1-2 箇所 = 30% → 10% で「本当に大事なところ」 だけ太字、強調機能を回復
3. リファレンス 1 本完成 → 残りは段階移行 = Phase 1 で運用境界確立 (= 新規は新フォーマット / 過去は子 Issue で段階移行)、全件一気リライトは MVP 過剰

### 副次知見 1: 後輩教材性 = 10 秒把握 + 精読の二層化

> ぜんぶ忘れていい、この 1 シーンだけ。
> 後輩は最初の 10 秒で「読む価値あるか」 判定する。だから **H2 直下に 3 行サマリ** を置く = 10 秒で核心が伝わる。詳細は H3 以降で精読する人向け。

dev-log のオンボーディング失敗パターン:
- パターン A: リード 3 段落の超長文で 10 秒判定不能 → 後輩離脱
- パターン B: 副次知見全部読まないと核心不明 → 後輩諦め

→ H2「教訓ハイライト」 直下に副次知見 3 件の 10 秒サマリ追加 (= 提案 1) + 各副次知見冒頭に 1 シーン比喩 (= 提案 2) で **「10 秒で核心、興味あれば精読」** の動線確立。

教材は「全部読む」 前提ではなく「興味のある層が読み深める」 前提で構成。

### 副次知見 2: 太字密度の上限ルール = 各段落 1-2 箇所

> ぜんぶ忘れていい、この 1 シーンだけ。
> 全部太字は **何も太字でないのと同じ**。「ここだけ大事」 を 1 つに絞ると太字が機能する。

day-11-4.md 初版 = 177 行中 51 行に `**` = 太字密度 28.8%。design-reviewer C-2 指摘「強調機能不全」 の典型。

リライト後 = 194 行中 15 行 = 太字密度 7.7% (= 3 者レビュー指摘吸収で bullet/番号付きリスト冒頭太字を全削除した 2 回目の実測値、本ファイル day-13-3.md も同連動修正で 7-8% 程度)。**ルール**: 段落 1 つにつき太字 1-2 箇所 + 列挙はリスト化 (= 太字なし) + 「絶対覚えてほしい 1 概念」 のみ太字。

→ 太字 = 「強調」 ではなく「**この 1 文だけは取り出して読んで**」 という指示。乱用は指示の信頼性を下げる。

### 副次知見 3: リファレンス 1 本完成 → 段階移行で MVP 過剰回避

> ぜんぶ忘れていい、この 1 シーンだけ。
> **全部一気にやらない**。1 本完成で「これが正解形」 を確定 + 残りは別 Issue で段階的に移行 = スコープ管理の基本。

Issue #279 当初の提案は「dev-log フォーマット改善」 だったが、20 ファイル全件リライトは 1 PR 規模超え (= 軽量 Issue 1 PR ルール違反)。

→ 3 フェーズ計画で本 PR (= Phase 1) はリファレンス 1 本 (= day-11-4.md) のみ確定:

- Phase 1 (= 本 PR): リファレンスリライトで「正解形」 確定
- Phase 2 (= 別 Issue、子起票候補): 過去 dev-log 1 ファイル 1 PR で段階的書き換え
- Phase 3 (= 別 Issue、子起票候補): dev-log-merge Skill のスケルトン同期

→ Phase 1 完了時点で **「新規 dev-log は新フォーマット使用、過去 dev-log は段階移行」** という運用境界が即時確立 = 後続 dev-log (= day-13-3.md 本ファイル!) は新フォーマットを採用済 (= 自己適用)。
