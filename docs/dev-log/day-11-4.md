### 副次知見 1: 学び 36 観測モード = 偶発観測 → 意図観測の新フェーズ

> ぜんぶ忘れていい、この 1 シーンだけ。
> 「**発火**」 = 自分が書いた言い切りに対して後続のレビューが「それ本当?」 と突っ込んでくること。
> 「**観測モード**」 = その突っ込みが来るか来ないかを意図的に試す実験 = 計画的に同じ言い切りを 5 箇所反復しておいて、レビューで再相対化指摘が出るかを観測。
> (詳細は [day-11.md](day-11.md) で確立 = 学び 36、memory `feedback_self_proposal_relativization.md` 正本 (= リポジトリ外、ローカル `~/.claude/projects/.../memory/` 配下のため GitHub UI からは参照不可))

Day 11-3 までは **偶発観測** (= 「気付いたら 3 箇所以上反復していた」 → 後付け抽出でサブパターン候補仮説化)。Day 11-4 で **意図観測モード** に到達 = strategic-reviewer 論点 3 を局長 OK で採用、「**本 PR は 2 mode で確定 (= 3 種類目 mode は v1.2 以降の検討)**」 を 5 箇所反復で計画的に発火装置を仕掛けた。

#### 5 箇所反復の配置

1. JS comment (= RECOVERY_LABELS 上): `native_health_controller.js` 冒頭、ラベル定数の用途説明として配置
2. JS comment (= recoveryAction 上): 同ファイルのメソッド直前、3 mode 拡張時の昇格検討メモとして配置
3. view comment (= recoveryButton マークアップ上): `show.html.erb`、SSR 用ダミー文言と組で配置
4. spec comment (= Issue #273 assertion 上): `settings_spec.rb`、3 mode 化時の assertion 追記想定として配置
5. PR description Summary: PR #277 本文、設計判断ログとして配置

#### 発火条件検証の論理構造

学び 36 サブパターン候補「3 箇所以上の同文反復 = 発火確率上昇」 を単発実例で確証することは原理的に不可能 (= 確証性向の罠)。反証は 1 件でできる (= 5 箇所反復しても発火しなければ仮説却下方向):

- 発火確認 → 仮説支持データ +1
- 不発継続 → 仮説却下方向 (= 「3 箇所以上反復」 でも発火確率上昇しない可能性)

いずれにしても観測価値あり = 仮説検証フェーズの正しい運用。

### 副次知見 2: 「観測モードに乗せない」 判断軸 = 自己提唱の言い切りに限る

> ぜんぶ忘れていい、この 1 シーンだけ。
> **観測装置に何を放り込むかの線引き**。Issue 起票時に自分で書いた「言い切り」 だけが発火対象、レビューで他人から指摘された派生は別 Issue に逃がす。混ぜると「装置が何を計っているか」 分からなくなる。

3 者並列レビュー指摘 9 件のうち、design I-1 (= retry 連続失敗時の UX ガード) のみ別 Issue (#276) 切り出し、残り 8 件は本 PR 吸収。判断軸は「スコープ内 / スコープ外」 の 1 軸で機械的に行われた:

- 本 PR 吸収 8 件 = code 4 + strategic 3 + design C-1 (= 文末削除)、いずれもスコープ内 (= 退化なし、文言・コメント・1 ファイル内修正で完結)
- 別 Issue 切り出し 1 件 = design I-1 (= retry 連続失敗時 UX ガード、retryCount 状態管理 + 文言切替 + リセット 3 タイミング)、スコープ外 (= 状態管理追加 + 仕様拡張で 1 PR 規模超え)

特に重要なのは **I-1 を「観測モードに乗せなかった」 判断**。観測モードは本 PR で確定する設計判断 (= 2 mode 化) に対する反復で発火装置を仕掛けるが、I-1 は本 PR 起票時に 1 文も触れていない外部観点 = 自己提唱なし = 学び 36 発火条件外。

観測モード乱用 (= レビュー由来の派生 Issue を全部本 PR に詰め込む) は発火条件を曖昧化するため、「**Issue 起票時に書いた言い切りに限り反復**」 という運用境界を本セッションで暗黙確立。

### 副次知見 3: 軽量 Issue 1 PR ルール常用フェーズ + Issue 削減 2 戦略軸

> ぜんぶ忘れていい、この 1 シーンだけ。
> 「**常用フェーズ**」 = 同じ手順が 4 回続くと「機械的に再現可能」 を超えて「考えずに使える」 段階に入ること。本 PR で 12 例目連続 = もう「やってる」 から「やってる前提」 になる。

Day 11-3 末の 9-11 例目 (= PR #270 / #271 / #272) から Day 11-4 で 12 例目達成 = 連続達成 4 件 + 累計 12 例目で **「機械的に再現可能」 を超えて「常用フェーズ」** 到達。

#### Issue 削減 2 戦略軸

局長指示「Issue を減らすことに注力」 に対する 2 つの戦略:

- A: 軽量から順消化 = 1 PR = 1 close で件数効率高、Closes 学習サイクル維持 (= 副作用: 大物 Tier 1 は残り続ける)
- B: Tier 1 大物攻略 = 大物 1 件で複数子 Issue が連動 close、教材性高 (= 副作用: 件数は一時増える)

本セッションは A 戦略採用 = 副局長判断で「Issue 件数を効率よく減らす」 文字通り解釈。Tier 1 大物 (= Phase 3 三重防衛 / Capacitor OAuth / daisyUI 全置換) は件数増やしうる性質を局長に提示し回避決定。**A vs B の判断軸自体が後輩教材**。

> **本判定 (= 「常用フェーズ到達」 / A 戦略採用) は観測モード非適用** = dev-log 内総括判定は観測対象外と扱う運用境界 (= 副次知見 2 で確立した「Issue 起票時の言い切りに限り反復」 ルールに基づく自己監査、strategic-reviewer 論点 1 B 案反映)。Day 12+ で本判定が再相対化された場合は別カウントで記録、観測モード発火件数とは混ぜない。
